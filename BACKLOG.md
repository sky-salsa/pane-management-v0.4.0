## Lessons Learned

| # | Description | Context |
|---|-------------|---------|
| L-09 | **Always verify persistence output exists** — tmux-resurrect was configured (plugins installed, continuum auto-save set to 15min) but never actually saved because the save directory was never bootstrapped. A 5-second `ls ~/.tmux/resurrect/` check would have caught this before the BSOD made it matter. After configuring any persistence mechanism, confirm the artifact exists on disk. | BSOD recovery — F-64 resurrect |
| L-08 | **Path normalization must be symmetric** — when deduplicating paths from two encoding systems (Windows `C:\` and WSL `/mnt/c/`), both branches of the normalizer must produce identical output. The dedup failed silently because one branch lowercased and the other didn't. Symmetric normalization = lowercase + canonical separators on ALL paths, not just the one you're converting. | F-63 dedup bug B-25 |
| L-07 | **Own your state, don't delegate to plugins** — tmux-resurrect saves tmux layout, but Pane Management already knows the layout (pane assignments, window names, grid geometry). Delegating to resurrect created a dependency on an external plugin that the app couldn't verify or control. The self-contained resurrect (F-64) uses the app's own settings.json and is fully testable. General principle: if you have the data, own the persistence. | BSOD recovery — F-64 resurrect |
| L-06 | **path_exists is unreliable from Windows Rust** — Windows Rust can't resolve WSL paths (`/mnt/c/...`), so `std::path::Path::exists()` returns false for many valid projects. Any feature using `path_exists` must verify via WSL (`getInode` or `wsl.exe stat`) before treating a path as truly gone. The F-61 inode system went through 3 fix iterations before this was properly understood. |
| L-05 | **Backfill + verify must not create false positives** — storing new metadata (inodes) can trigger UI conditions that weren't previously reachable. The Unlinked badge showed for all projects after inode backfill because the condition checked `inode exists + path_exists false`. Solution: use explicit flags set only by the verification pipeline, not inferred from partial state. |
| L-04 | **Don't hide data to fix bugs** — when resume was picking empty sessions, the instinct was to filter them from the list. User correctly pushed back: hiding data risks unpredictable downstream effects. The fix was making the auto-pick smarter (skip small files) while keeping everything visible. Rule: fix the selection logic, not the visibility. |
| L-03 | **Talk through the problem before coding** — the pane detection bug (B-16/B-17) went through several failed fix attempts (synthetic assignments, forced fallback buttons, etc.) before we stopped and traced the actual data flow: JSONL files → scanner → actual_path → comparison. The root cause was a WSL/Windows path format mismatch in the comparison, plus the scanner not reading deep enough. Talking through it with the user and examining real session files revealed this in minutes. Lesson: when fixes aren't working, stop coding and investigate the data. |
| L-02 | **Use standard CSS patterns for common UI** — spent too many iterations hand-tuning a toggle switch (knob sizing, alignment, positioning). The standard pattern is: track as a div, knob as `::after` pseudo-element, same height as track, `left: 0` / `left: (width - knob)` for on/off. Dispatching a quick research sub-agent (NOT a GSD researcher — those are slow) resolved it in seconds. For solved UI problems, look up the pattern first. | Always-on-top toggle pill CSS |
| L-01 | **Wrong component, not stale cache** — when a UI change doesn't appear after rebuild, check if there's a DIFFERENT component rendering the same view before blaming Vite cache. In this project, `ProjectDetailModal.tsx` renders sessions inline with its own buttons — it does NOT use `SessionItem.tsx`. The delete button was added to SessionItem but the modal uses its own markup. Rule: grep for the actual button text visible in the UI to find the real component. | F-29 delete session implementation |

## Architecture Notes

- **Windows/WSL project merge (F-63):** Discovery now auto-discovers ALL project directories (both `C--` and `-mnt-c-` prefixes). Projects are deduplicated by normalized `actual_path` — both variants merge into one sidebar entry with `-mnt-c-` as primary. `secondary_dirs` field on ProjectInfo carries the `C--` counterpart. On startup, `mergeWindowsWslProjects()` saves secondary dirs into `claude_project_dirs` for umbrella session loading. Sessions from `C--` dirs show a blue "Windows" pill. Resume of Windows-only sessions triggers a copy-to-WSL modal explaining the desync tradeoff. `copy_session_to_wsl` Rust command handles the file copy (JSONL + subagent subdirectory). Path normalization for dedup: both `/mnt/c/Users/...` and `C:\Users\...` lowercase to same key.
- Terminal abstraction trait supports Warp, PowerShell, and tmux backends. Only tmux actively used.
- Session binding logic: project defaults to most recent session, explicit binding overrides until unbound.
- **F-61 Inode identity system:** Projects tracked by filesystem inode (stable across renames). Stored in `project_meta.inode`. `claude_project_dirs` array links multiple Claude Code encoded directories to one project after renames. Reconciliation runs once on startup: backfills inodes, verifies paths via WSL stat, escalating scan for orphans. `path_exists` from Rust is unreliable for WSL paths — always verify via `getInode()`.
- **Theme system:** CSS custom properties with `data-theme` attribute. Theme persisted in settings.json, loaded in App.tsx module init before render. Theme-specific components (particles, sigils, vines) conditionally rendered via `isWitchingHour()` signal watching MutationObserver on document.documentElement.
- **Approval detection:** `check_pane_statuses` runs via 3s poll, captures last 10 lines of Claude-running panes via `tmux capture-pane`, checks for `❯ N.` pattern. Muting filters at the data level before storing in state.
- **Self-contained resurrect (F-64):** Resurrect button reads pane_assignments from settings.json, creates tmux session + windows + grids, cd's each pane, does NOT auto-launch Claude. Pane index remapping via visual position sorting. No dependency on tmux-resurrect.
- **State persistence (F-65):** settings.json now stores: `window_names` (session|window → name), `window_layouts` (session|window → {cols, rows}), `always_on_top` (boolean), `session_names` (project → {session_id → name}, migrated from localStorage). All read on mount, written on user action.
- **Windows/WSL project merge (F-63):** `list_projects` deduplicates by normalized actual_path (case-insensitive, separator-normalized). `-mnt-c-` is primary, `C--` goes into `secondary_dirs`. On startup, `mergeWindowsWslProjects()` populates `claude_project_dirs`. Sessions carry `source_dir` field for origin tracking. `copy_session_to_wsl` Rust command handles cross-encoding file copy.
